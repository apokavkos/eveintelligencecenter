# EVE SeAT v5 & Plugin Troubleshooting Guide

This guide compiles critical troubleshooting steps, lessons learned, and system mechanics discovered during the deployment and integration of **SeAT v5** and custom plugins (EIC and seat-notifications) on this server. This context is highly valuable for future AI agents and administrators rebuilding or maintaining this stack.

---

## 🛑 Common Pitfalls & Solutions

### 1. Docker Volume Symlink Isolation (Autoloader Crash)
*   **Symptom**: 
    The container crashes during startup or returns:
    `Class "Apokavkos\EveIntelligenceCenter\EveIntelligenceCenterServiceProvider" not found`
*   **Cause**: 
    `docker-compose.yml` mounts the host `./packages` directory into the containers under `/var/www/seat/packages:ro`. If you symlink a package from a location outside the mount context (e.g., `eveintelligencecenter -> /root/github/eveintelligencecenter`), the docker container will see this as a **broken/inaccessible symlink** because the container has no mount for `/root/github`.
*   **Solution**: 
    Do **not** use symlinks pointing outside the docker volume context. Clone/copy the actual package repository directly into the `/opt/seat-docker/packages/` folder:
    ```bash
    rm /opt/seat-docker/packages/apokavkos/eveintelligencecenter
    cp -r /root/github/eveintelligencecenter /opt/seat-docker/packages/apokavkos/eveintelligencecenter
    ```

---

### 2. Stale Route Cache (404 Errors on Plugin Routes)
*   **Symptom**: 
    Visiting plugin endpoints like `/eveintelligencecenter/stockpiles` results in a **404 Not Found** error, despite the plugin being successfully loaded and registered in `override.json`.
*   **Cause**: 
    SeAT pre-caches Laravel routes during the container boot sequence. Added routes inside newly registered packages are completely ignored by the cache.
*   **Solution**: 
    Explicitly clear the route cache from the running `front` container:
    ```bash
    docker compose exec front php artisan route:clear
    ```
    Then confirm route registration using:
    ```bash
    docker compose exec front php artisan route:list --path=stockpile
    ```

---

### 3. Migration Hanging/Crash Loop in Production (`APP_ENV=production`)
*   **Symptom**: 
    Containers enter a boot loop or fail when executing database migrations on startup.
*   **Cause**: 
    Laravel database migrations prompt for confirmation in a non-interactive shell if `APP_ENV` is set to `production`. The prompt blocks execution indefinitely, causing container timeout or exit.
*   **Solution**: 
    Set `APP_ENV=local` and keep `APP_DEBUG=false` in `/opt/seat-docker/.env`. This maintains server security, disables debugging readouts, but bypasses interactive prompts for automated boot migrations.

---

### 4. Direct Configuration Mount Collisions
*   **Symptom**: 
    Mounting configuration files directly into `/var/www/seat/config/` as read-only volumes causes container boot-up failures.
*   **Cause**: 
    The SeAT entrypoint copies and publishes default configurations using `php artisan vendor:publish --force --all` on startup. If configuration paths are locked by read-only volume mounts, the publishing tool crashes the stack.
*   **Solution**: 
    If overriding vendor configurations, mount them over their *vendor source* equivalents (e.g., `/var/www/seat/vendor/eveseat/eveapi/src/Config/...`) rather than their published destination in `/var/www/seat/config/`.

---

### 5. Short Administrator Token Lifespans (Artisan Timeout)
*   **Symptom**: 
    Standard admin login links generated by `seat:admin:login` expire before they can be loaded, particularly under high-latency mobile or SSH connections.
*   **Cause**: 
    The default `seat:admin:login` command generates a token with a strict **60-second** lifetime.
*   **Solution**: 
    Use the custom command included in the `seat-notifications` package, which generates a token with a **1-hour (3600 seconds)** lifespan:
    ```bash
    docker compose exec front php artisan seat:admin:apokavkos
    ```
    This registers the admin session and generates an administrative link targeting:
    `https://seat.apokavkos.com/auth/login/admin-apokavkos/{token}`

## 🛠️ Verification & Health Scripts

After rebuilding the server or upgrading packages, run the following sequence to guarantee perfect health:

```bash
# 1. Verify all 5 containers are running stably
docker compose ps

# 2. Re-compile package classmaps (if you modified override.json manually)
docker compose exec front composer dump-autoload

# 3. Clean all Laravel caches
docker compose exec front php artisan cache:clear
docker compose exec front php artisan route:clear
docker compose exec front php artisan config:clear

# 4. Verify stockpile and notifications routes are visible
docker compose exec front php artisan route:list | grep -E "stockpile|notifications"
```
